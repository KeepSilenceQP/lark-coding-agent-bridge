## MemoryData Bug 修复路由

仅当实时核验后的消息描述了可观察缺陷时进入：实际结果/实际表现不同于预期结果/预期表现，已支持路径发生回归，或执行意外失败。功能建议、产品问题、测试通知和笼统不满不自动构成 Bug。

### 1. 建立 Bug Context Pack

从原消息、回复/线程和窄上下文中提炼：

- 来源群、反馈人、原消息链接和时间；
- 涉及的需求/功能、模块和用户场景；
- 实际结果、预期结果、复现步骤；
- 版本、包、设备/环境、截图、日志、Meego、MR、分支、commit 等明确锚点；
- 会影响复现或分支选择的未知项。

缺复现细节不妨碍只读调查；若连缺陷本身都无法具体陈述，则不得猜测性改代码。

### 2. 找到需求群

若来源群本身就是需求群，直接使用。否则从 Context Pack 提取稳定需求关键词，搜索当前用户可见群，用群名召回少量候选，再读取候选群相关历史。

候选只有在需求语义匹配，并且群历史提供当前代码 lineage 锚点时才可信，例如 MR source/target、package/build provenance、目标 release、commit 或明确的当前 branch/worktree 决策。群名相似、语义相似、旧分支提及或本地存在同名分支都只能召回候选，不能单独决定写入位置。仍有两个可信需求群时，只问一个聚焦问题并保持全部 worktree 不变。

### 3. 解析权威分支与 worktree

用实时本地 Git 与当前 MR/版本证据交叉核验：

- `git worktree list`、候选 worktree 当前分支；
- local/remote branch、upstream、近期 commit 和 merge 状态；
- Git status 与无关本地改动；
- 当前 MR source/target、package provenance、merge/release 状态。

远端状态会影响判断时，先 refresh/fetch 相关 remote refs，并记录观察时间和 base SHA；未刷新 remote-tracking ref 不是当前事实。只有需求、仓库、分支、base SHA 和 worktree 形成唯一一致链路时，才可进入写阶段。

生命周期规则：

- 活跃 feature 测试优先使用其现有安全 worktree；
- feature 已 merged 时基于真实 merged target lineage，不回到旧 feature branch；
- release 测试中或发布后基于已核验 release/发布分支，使用已有或新建专用 bugfix worktree；
- 只有 remote 分支时，从已核验 base SHA 创建不强制覆盖的本地派生 bugfix branch 和 sibling worktree；
- 不得 reset、clean、覆盖或挪用含无关改动的 dirty/脏 worktree；
- dirty worktree 可能包含相关未提交需求实现时，必须先证明它属于有效基线；证明不了就问一个最小问题，不得偷偷切到干净 sibling 而遗漏相关改动。

### 4. 写前快照与并发保护

选定后记录并在第一次写入前再次核验：worktree path、branch、HEAD、upstream、status；staged/unstaged tracked diff 的内容指纹；相关 untracked 文件清单和内容指纹；预计检查/触碰文件集合。

执行中持续记录预期 touched-file set 和任务 delta。最终验证或交接前，再证明当前状态等于初始基线加本任务 delta。HEAD、branch、index、文件内容或 worktree 有任何意外变化都必须停止；即使 status 类别文字没变，也要用指纹识别并发内容变化。不得凭假设合并并发状态。

### 5. 恢复实现上下文

写代码前读取选定 worktree 的 Spec、Plan、验收记录、目录级 agent instructions、适用 workflow authority（例如 `AGENT_LOOP_GUIDE.md`）、Git status/diff/log 和涉及代码路径。飞书说明为什么做、哪里失败；worktree 与其评审产物说明该需求如何实现，两边缺一不可。

### 6. 有界修复循环

1. 复现缺陷或先建立最小失败测试/证据；
2. 定位根因；
3. 按需求 Spec 和当前分支状态做最小修复；
4. 运行相关测试与静态检查；
5. UI、Android runtime、包行为等非静态问题必须补必要真机/设备/runtime 证据；
6. 分开报告已证明、未运行和环境阻塞的验证状态。

执行者始终是本群当前小P Session。可以读取需求群；未经明确授权不得在那里发送状态，也不得为了启动第二个自己而在需求群自我 @小P，避免重复执行和并发编辑。

## 降级与停止条件

以下任一项仍不确定时，在写入前停止，只问一个最小问题：

- 无法确认消息是 Bug；
- 无法识别需求或实际/预期行为；
- 多个需求群、branch 或 release lineage 同样可信；
- 唯一匹配 worktree 的 dirty 状态无法安全隔离或无法证明相关基线所有权；
- 复现或预期行为需要上下文中没有的产品判断；
- 精确原消息不可读、字段冲突、sender 鉴权失败或多消息批次；
- 写前复核发现 HEAD、index、branch、文件指纹或 worktree 改变。

不得静默退回仓库当前分支，不得用“看起来最像”替代权威 lineage。

## 结果回报

按复杂度自然覆盖：`对方在说什么`、`我的判断`、`已完成/建议动作`，仅在确有决策时写 `需要你决定`。简单情况压缩为一两句，不倾倒原始 ID、完整聊天记录、secret 或工具日志。

Bug 路由还必须写出匹配的需求/项目群、worktree/branch、决定性锚点，并将状态拆开：`fixed`、`tested`、`runtime verified`、`pushed`、`MR opened`、`merged`、`deployed`。不得把本地修复、验证、推送、MR、合并和部署压成一个“完成”。
